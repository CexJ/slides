Qu'est-ce que la programmation fonctionnelle ?

Un peu d'histoire

  • (∼1930) Alonzo Church introduit le λ-Calculus
  • (∼1930) Alan Turing introduit la Machine de Turing
  • (∼1936) Alonzo Church et Alan Turing prouvent l'équivalence des deux théories
  • (∼1940) John von Neumann introduit l'architecture des ordinateurs modernes

La programmation impérative descend de la machine de Turing tandis que la programmation fonctionnelle descend du λ-Calculus

La programmation fonctionnelle

Sépare le code pur du code impur

Les actions impures sont :

  • La lecture d'une source d'une source externe (fichiers, base de données, internet...)
  • L'écriture sur une ressource externe (logs, fichiers, base de donnes, internet...)
  • La mutation de la valeur d'une variable

La programmation fonctionnelle "pure"

Élimine virtuellement le code impure

Pourquoi la programmation fonctionnelle ?

La transparence référentielle

La transparence référentielle est une propriété des expressions d'un langage de programmation qui fait qu'une expression peut être remplacée par sa valeur sans changer le comportement du programme

Exemple

Integer x1 = reference.getX(param);
Integer x2 = reference.getX(param);
Integer sum =  x1 + x2;	
							
Integer sum = 2*reference.getX(param);

Le code pur est plus facile à tester

Sans dépendances externes et états mutables les tests sont plus fiables.

Le code est plus expressif et plus facile à lire

Dans un langage fonctionnelle statiquement et fortement typé, un peut exprimer les contraintes directement dans les types.

Exemple

Integer maxCopiesInEn(List<Book> books){
  Integer res = null;
  for(Book book : books) {
    if("en".equals(book.getLanguage())) {
      Integer bookNumCopies = book.getNumCopies()
      if(res == null || res < bookNumCopies) {
        res = bookNumCopies;
      } 
    }
  }
  if(res == null) {
    throw new RuntimeException();	
  } else {
    return res;
  } 
}

En programmation fonctionnelle

Optional<Integer> maxCopiesInEn(List<Book> books){
  return books.stream()
           .filter(book -> "en".equals(book.getLanguage()))
           .map(Book::getNumCopies)
           .max((x,y) -> x < y);
}

La programmation concurrente / parallèle / distribuée est plus facile

Sans états mutables, pas besoin de blocs de synchronisation de sorte que les fonctions pures sont bien adaptées pour décrire des tâches à exécuter en parallèle.

... et pourquoi maintenant ?

Si la programmation fonctionnelle est bonne, pourquoi ne pas l'avoir utilisé avant ?
  • Il faut utiliser un Garbage Collector
  • Pour avoir de bonnes performances il faut avoir un bon compilateur / interpréteur
  • Elle brille avec une complexité et une abstraction plus élevées
  • Elle brille dans la programmation concurrente / parallèle / distribuée
  • Elle est plus difficile à écrire

Aucune de ces raisons n'est plus valable aujourd'hui
