Cos'è git?

Wikipedia

Git è un software di controllo versione distribuito utilizzabile da interfaccia a riga di comando, creato da Linus Torvalds nel 2005.

Si tratta di un software che gestisce automaticamente la storicizzazione ed il versionamento dei file contenuti in una cartella.

Persistent List (parte I)

List

Per imparare ad usare Git bisogna capire che struttura di dati astratta utilizza al suo interno.

Una lista è una sequenza finita, ordiata e omogenea di elementi.

Ordinata: gli elementi sono ordinati con un indice.

Omogenea: gli elementi sono dello stesso "tipo".

In una normale lista possiamo:

  • Aggiungere un elemento in coda
  • Leggere l'elemento all'indice i
  • Modificare l'elemento all'indice i
  • Eliminare l'elemento all'indice i

Persistent

Una lista persistente le operazioni possibili sono solamente:

  • Aggiungere un elemento in coda
  • Leggere l'elemento all'indice i, ottenendone una copia

Storicizzazione

Per la storicizzazione le liste persistenti sono ottime, le passate versioni non devono mai essere modificate.

L'elemento in coda rappresenta l'elemento attivo.

Se si vuole ritornare ad una versione precedente d'indice i basta leggere l'elemento e aggiungerlo alla coda della lista.

Puramente funzionale

Come vedremo in seguito è importante capire che queste liste sono puramente funzionali.

In una lista classica le operazioni modificano la lista stessa: se ho una lista \(l\) di \(n\) elementi, se ne elimino uno \(l\) sarà una lista con \(n-1\) elementi.

In una lista puramente funzionale le operazioni sono funzioni che creano una nuova lista: se ho una lista \(l_1\) di \(n\) elementi, se invoco la funzione per eliminarne uno otterrò una nuova lista \(l_2\) con \(n-1\) elementi, mentre la prima lista \(l_1\) resterà invariata.

Storicizzare i documenti

La branch master

Git lavora con le cartelle, data una cartella potrà gestirà il versionamento dei file presenti in una cartella (ed in tutto l'albero delle sotto cartele).

Una volta installato git l'operazione per fargli gestire una cartella è git init, la cartella ed i suoi file rimarranno invariati e funzioneranno come prima, l'unica differenza sarà la creazione di un file .git

La creazione, la modifica o il salvataggio di un file non implicano automaticamente un versionalento di git. Questo deve essere domandato esplicitamente.

Se si crea un file che si vuol far gestire da git bisogna utilizzare il comando git add

Se si modifica o elimina un file gestito da git e si vuole storicizzare tale cambiamento bisogna utilizzare il comando git commit, aggiungendo un messaggio per indicare la motivazione di tale cambiamento.

Queste operazioni corrispono ad aggiungere un elemento ad una persistent list che prende il nome di master branch

Se si vuole ottenere una copia di un precedente elemento (commit) bisogna utilizzare il comando git revert.

Persistent List (parte II)

Una lista è un ramo di un albero

Come già detto precedentemente una lista persistente è una struttura di dati puramente funzionale.

Sia \(l\) una lista, se aggiungiamo un elemento a questa lista otterremo una nuova lista \(l_1\) lascianod \(l\) invariata.

Se aggiungiamo un altro elemento a \(l\) una nuova lista \(l_2\).

Le liste \(l_1\), \(l_2\) e tutte coloro che creeremo aggiungendo nuovi elementi sono tutte diverse ma condividono il segmento \(l\), sono i rami di un albero.

L'albero delle versioni

Questa struttura ci è utile per descrivere le diverse versioni di uno stesso documento che possono essere modificate in parallelo.

Versionamento dei documenti, le altre branch

Muoversi tra le branch

Per spostarsi da una branch all'altra bisogna utilizzare il comando git checkout.

Nel caso in cui questa branch non esiste già verrà creata a partire dalla versione in cui si sta lavorando.

Modificare il passato

Il comando git checkout può essere utilizzato per spostarsi su un commit preciso di una branch, ma se questo non è in posizione head non sarà possibile effettuare dei nuovi commit (bisogna ricordarsi che la nostra lista è immutabile e non possiamo cambiare il passato).

Sarà però possibile creare una nuova branch, in questa branch saremo in posizione head e potremo dunque effettuare dei nuovi commit.

L'avvantaggio delle versioni

Utilizzare diverse versioni e farle avanzare in parallelo è la funzionalità centrale di un gestore di versioni e permette d'aumentare considerevolmente la propria produttività.

Per rendere veramente utilizzabile tale funzione è però necessaria un'operazione particolare che permetta di riunificare due branch separate.

Esempio

Supponiamo di lavorare su un documento, potremmo usare la branch master per gestire le versioni stabili.

Ogni qualvolta volessimo modificare un capitolo potremmo aprire una nuova branch.

In questo modo potremmo avere sempre una versione stabile (master) e varie versioni provvisorie su cui lavorare in parallelo.

Una volta che avremo finito le modifiche sulle branch vorremmo riportarle sulla versione stabile (master).

L'operazione merge

Per poter riportare le modifiche di una branch su un'altra utilizzeremo il comando git merge.

Questa operazione sarà automatica se una delle due branch èun segmento dell'altra.

Nel caso le due branch siano ben distinte l'operazione di merge potrebbe non essere risolvibile in modo automatico e ci troveremo in presenza di conflitti.

Per poter terminare l'operazione di merge bisognerà risolvere questi conflitti e fare un commit di tale soluzione.

Persistent List (parte III)

Le liste persistente in un contesto concorrente

Una delle proprietà che derivano dall'imutabilità delle liste persistenti è la facilità con cui possono essere utilizzate in un contesto di calcolo concorrente.

Dati due attori che i due possono condividere una lista persistente su cui entrambi lavoreranno.

Ogni operazione creerà una nuova lista, locale all'attore che l'ha compiuta, senza modificare la lista di partenza e quindi senza creare conflitti con il secondo attore.

Queste strutture di dati possono essere utilizzate in un sistema distribuito, dove ogni nodo avrà una copia locale dell'albero delle versioni e delle operazioni simili ad un merge locale renderanno possible la sincronizzazione.

Decentralizzazione e collaborazione

Decentralizzazione

Git è un programma decentralizzato, il che implica che funziona in un cluster di nodi collegati tra loro.

Ogni nodo ha una copia del repo git.

Creazione di un nuovo nodo

Dato un nodo con un repo git un altro nodo potrà entrare a far parte del cluster utilizzando il comando git clone.

Il repo git del nuovo nodo sarà inizializzato con la copia del repo git del nodo remoto indicato come parametro.

Pull

Una volta in possesso di un repo git sarà possibile lavorarci localmente con le operazioni già citate.

Per resincronizzare il repo locale con il repo git d'origine si potrà utilizzare il comando git fetch.

Questo comando copierà il repo remoto nell'head.

Per resincronizzare la workdirectory con il repo git d'origine si potrà utilizzare il comando git pull.

Questo comando copierà il repo remoto nell'head ed eseguirà un merge con la workdirectory, se ci fossero dei conflitti per concludere il pull sarà necessario risolverli e di seguito eseguire un commit.

Push

Se si vuole sincronizzare un nodo remodo con il proprio repo locale head si può utilizzare il comando git push.

Questa operazione sarà bloccata in presenza di conflitti, bisognerà risolverli prima di poter completare l'operazione.

Collaborazione

Git è un programma decentralizzato, ovvero non esiste il concetto di nodo master (server centrale) ma ogni nodo e pari agli altri.

Questo è un modello molto più flessibile di quello centralizzato, che può essere simulato in un contesto decentralizzato eleggendo un nodo a master.

GitHub

Una topologia classica è quella di utilizzare un repo remoto passivo come server centrale per sincronizzare diversi nodi.

Ci sono vari siti/servizi online che permettono d'aver un repo git a tale scopo. Uno dei più utilizzati è github.
