Motivations

Naive approach in parsing

The naive approach to parsing an input: extracting and validating by hand the values, has some shortcomings:

  • Over complicated logic hard to refactor.
  • Hard to maintaining, input formats change often.
  • Ad hoc solution must be created for each entry: repeated code or unwanted dependencies.

An abstract approach

For example, let suppose we need to parse various csv documents.

With the naive approach we create an algorithm for each different kind of csv document. A more abstract approach is to write an algorithm to read csv documents and to configure it for each document kind.

The idea is to refactor some commons pattern into some parser, an object that can be configured to parse any kind of csv.

A more generalized approach

What has in particular a format from another?

The idea is to create a framework that we can use to implement generalized parser for each kind of format we want.

What declarative means?

The ingredient for a parser

To make a parser works we need to configure it for the specific kind of input we want to parse.

This configuration is a description and it can take various form.

Declarative vs imperative

A description is by nature of declarative form since it declares some kind of shape.

In opposite of imperative style where we list the actions we need to follows to parse something.
